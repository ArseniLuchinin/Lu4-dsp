# Виртуальная передача данных (VirtualTransmitter)

## Обзор

`VirtualTransmitter` — механизм **широковещательной рассылки данных между конвейерами**. Он позволяет одному конвейеру отправить обработанные данные, а другим конвейерам — получить их по именованному тегу. Это единственный способ обмена данными между конвейерами: внутри одного конвейера модули передают `shared_ptr` напрямую, а между конвейерами — через именованные каналы.

Ключевые особенности:
- **Статический брокер** — один на всё приложение (singleton-подобный подход через статические члены)
- **BroadcastSlot** — ячейка хранения данных для конкретного тега
- **Оптимизация копирования** — первые N-1 получателей получают копию, последний забирает оригинал через `std::move`
- **Блокирующее ожидание** — при использовании `@tag` в `pipeline.json` инициализация модуля блокируется до получения данных
- **Таймаут 45 секунд** — при превышении процесс завершается через `exit(1)`

## Архитектура

### BroadcastSlot

Каждый тег имеет слот, хранящий:

| Поле | Тип | Назначение |
|---|---|---|
| `data` | `shared_ptr<IData>` | Данные для рассылки (`nullptr`, когда все получили) |
| `expectedReceivers` | `size_t` | Ожидаемое количество получателей (авто-подсчёт через `registerRx`) |
| `deliveredCount` | `size_t` | Сколько RX уже забрали данные |
| `iteration` | `size_t` | Номер итерации рассылки |
| `txRegistered` | `bool` | Зарегистрирован ли отправитель (один тег = один TX) |
| `rxCv` | `condition_variable` | Уведомление о получении данных и освобождении слота |

### VirtualTransmitter

Реализован через статические члены:

| Поле | Тип | Назначение |
|---|---|---|
| `s_broadcastSlots` | `map<string, BroadcastSlot>` | Все слоты по тегам |
| `s_receiverCounts` | `map<string, size_t>` | Счётчики зарегистрированных RX (для авто-подсчёта) |
| `s_mutex` | `mutex` | Глобальная блокировка для `s_broadcastSlots` и `s_receiverCounts` |
| `s_timeoutMs` | `atomic<int>` | Таймаут ожидания — 45 000 мс |

Почему статический брокер? Потому что конвейеры работают в разных потоках, но в одном процессе. Статические члены доступны из любого потока без необходимости передачи указателя. Это упрощает интерфейс: модулю не нужно знать, где находится брокер, он просто создаёт локальный объект `VirtualTransmitter` и вызывает его методы. Все экземпляры `VirtualTransmitter` работают с одним и тем же статическим состоянием.

## Методы

### `registerTx(tag)`

Регистрирует отправителя для тега. Возвращает `false`, если для этого тега уже зарегистрирован TX. Это гарантирует, что один тег имеет ровно одного отправителя, что упрощает семантику: не нужно разрешать конфликты при одновременной записи от нескольких TX.

### `registerRx(tag)`

Инкрементирует счётчик получателей для тега. Вызывается автоматически при разрешении `@tag` в `fetchParam` или через `IVirtualRX::setTag()`. Важно, что регистрация происходит на этапе сборки конвейера, а не во время выполнения. Это позволяет TX узнать, сколько копий данных нужно подготовить, ещё до первой отправки.

### `txData(data, tag)`

**Отправка данных.** TX блокируется, только если предыдущая итерация ещё не завершена (не все RX забрали данные). При превышении таймаута ожидания освобождения слота процесс завершается через `exit(1)`. Затем записывает данные в слот, сбрасывает `deliveredCount = 0`, инкрементирует `iteration` и вызывает `notify_all()` для разблокировки ожидающих RX.

Блокировка отправителя редка в нормальных условиях: она возникает только если TX работает быстрее, чем RX успевают забирать данные. Это защита от переполнения: без неё слот бы накапливал неограниченное количество устаревших данных. Разблокировка TX происходит, когда последний RX забирает оригинал данных через `std::move` и вызывает `notify_all()`.

### `waitRxData(tag)`

**Блокирующее получение данных.** Ждёт появления данных через `condition_variable::wait_for`. При таймауте — вызывает `exit(1)` (жёсткое завершение процесса).

**Оптимизация копирования:** если `deliveredCount < expectedReceivers - 1`, метод возвращает `data->copy()` — глубокую копию через `cudaMemcpyDeviceToDevice` для GPU-данных. Если это последний получатель, возвращает `std::move(slot.data)` — оригинал без копирования. После этого `slot.data` устанавливается в `nullptr`, слот освобождается для следующей итерации. Кроме того, последний RX уведомляет ожидающий TX через `notify_all()`.

Эта оптимизация критически важна для производительности. Представим, что один конвейер вычисляет коэффициенты FIR-фильтра и рассылает их трём другим конвейерам. Без оптимизации понадобилось бы 3 полных копирования GPU-памяти. С оптимизацией — только 2. Для больших массивов (мегабайты коэффициентов) это заметная экономия пропускной способности памяти.

Почему копирование вообще необходимо? Потому что каждый RX-получатель имеет право модифицировать данные. Например, конвейер с FIR-фильтром может дописать историю в буфер коэффициентов. Если бы все получатели ссылались на один и тот же `shared_ptr`, изменения одного конвейера повлияли бы на другой.

### `rxData(tag)`

**Неблокирующее получение.** Возвращает `nullptr`, если данных нет. Использует ту же логику copy/move. Полезен, когда модуль хочет проверить наличие данных, не засыпая.

### `checkData(tag)`

Проверяет наличие данных в слоте. Возвращает `bool`. Используется для диагностики и отладки.

### `setTimeoutMs(ms)`

Устанавливает таймаут ожидания в миллисекундах. По умолчанию 45 000 мс (45 секунд). Применяется ко всем последующим операциям `waitRxData` и блокирующим `txData`.

### `getTimeoutMs()`

Возвращает текущий таймаут ожидания в миллисекундах.

### `findTeg(name)`

Устаревший метод, заменён на `registerTx`. Проверяет наличие тега в слотах. Объявлен как `[[deprecated("Use registerTx instead")]]`.

## Модули VirtualTX и VirtualRX

### VirtualTX

**Роль:** Конечный модуль конвейера-отправителя.

**Параметры:** `tag` (string).

**Логика:** При `setParam("tag")` вызывает `registerTx(tag)`, резервируя право отправки. При `setData()` принимает данные от предыдущего модуля. При `run()` отправляет их через `txData(data, m_tag)`. `init()` проверяет, что тег не пуст. Деструктор отправляет `EmptyContainer` как сигнал завершения — это позволяет RX-получателям узнать, что отправитель больше не будет слать данные.

VirtualTX удобен, потому что он выглядит как обычный модуль для Conveyor. Конвейер не знает, что данные уходят в другой конвейер — он просто вызывает `run()` и `getData()` как для любого другого модуля. А VirtualTX внутри решает, куда отправить результат.

### VirtualRX

**Роль:** Начальный модуль конвейера-получателя (source).

**Наследует:** `IVirtualRX`.

**Логика:** `IVirtualRX::setTag()` автоматически вызывает `registerRx(tag)`. `init()` проверяет, что тег не пуст. При `run()` вызывает `rxData()` (внутри `waitRxData`), сохраняет в `m_data`. `getData()` передаёт данные следующему модулю в цепочке. `setData()` не поддерживается (возвращает `false`).

VirtualRX удобен тем, что он скрывает всю сложность блокирующего ожидания. С точки зрения Conveyor это обычный источник данных, как `FileSrc`. Conveyor просто вызывает `run()` и `getData()`. А внутри VirtualRX может происходить ожидание данных от другого конвейера, которое длится секунды.

## Синтаксис @tag в pipeline.json

В `pipeline.json` для ссылки на данные из другого конвейера используется синтаксис `@tag_name`:

```json
{
    "name": "FIR-filter",
    "params": { "taps": "@fir_rrc_coeff" }
}
```

### Как это работает

Разрешение происходит в `IModule::fetchParam()` (`Module/src/IModule.cpp`):

```cpp
const auto tagToken = getTagToken(value);  // строка начинается с @
if (!tagToken.empty()) {
    VirtualTransmitter transmitter;
    const std::string tag = tagToken.substr(1);
    transmitter.registerRx(tag);              // регистрируем как RX
    auto rxData = transmitter.waitRxData(tag); // БЛОКИРУЕТСЯ!
    setParam(paramName, std::any(rxData));    // передаём данные модулю
}
```

**Важно:** при сборке конвейера `fetchParam` **блокирует инициализацию модуля**, ожидая данные от другого конвейера. Это создаёт неявную синхронизацию между конвейерами на этапе старта.

Почему это удобно? Потому что разработчику не нужно вручную управлять порядком запуска конвейеров. Conveyor B может начать сборку раньше, чем Conveyor A вычислит коэффициенты. Модуль с `@tag` просто подождёт. Это устраняет race condition: RX никогда не получит неинициализированные данные, потому что он не начнёт работу до фактического получения.

## Пример потока данных

**Конвейер A (вычисление коэффициентов):**
```json
{
    "name": "rrc_compute",
    "modules": [
        { "name": "RRCCompute", "params": { ... } },
        { "name": "VirtualTX", "params": { "tag": "fir_rrc_coeff" } }
    ]
}
```

**Конвейер B (обработка сигнала):**
```json
{
    "name": "signal_process",
    "modules": [
        { "name": "FileSrc", "params": { "file name": "..." } },
        { "name": "FIR-filter", "params": { "taps": "@fir_rrc_coeff" } },
        { "name": "..." }
    ]
}
```

### Порядок выполнения

1. **Сборка конвейера B:** при обработке параметра `taps = "@fir_rrc_coeff"`, `FIR-filter` вызывает `registerRx("fir_rrc_coeff")` и блокируется в `waitRxData("fir_rrc_coeff")`.
2. **Конвейер A запускается:** `RRCCompute` вычисляет коэффициенты.
3. **Отправка:** `VirtualTX::run()` вызывает `txData(data, "fir_rrc_coeff")`.
4. **Разблокировка:** `waitRxData` в конвейере B получает данные, блокировка снимается.
5. **Продолжение:** конвейер B продолжает работу с полученными коэффициентами.

Такой поток данных типичен для реальных пайплайнов: один конвейер готовит «медленно меняющиеся» данные (коэффициенты фильтра, справочные таблицы), а другие используют их многократно. Блокирующее ожидание гарантирует, что «быстрый» конвейер никогда не начнёт работу с устаревшими или отсутствующими коэффициентами.

## Оптимизация копирования

При широковещательной рассылке `VirtualTransmitter` минимизирует копирование. Первые N-1 получателей получают глубокую копию через `data->copy()`, а последний — оригинал через `std::move(slot.data)`. После того как все `expectedReceivers` забрали данные, `slot.data` становится `nullptr`, и слот готов к следующей итерации.

Это эффективно, потому что экономит одно копирование из N. Для GPU-данных копирование — это `cudaMemcpyDeviceToDevice`, которая хоть и быстрая, но всё равно потребляет пропускную способность памяти. Когда данные рассылаются трём и более получателям, экономия становится существенной.

Кроме того, освобождение слота (`slot.data = nullptr`) происходит автоматически, как только последний получатель забрал данные. Это предотвращает утечки: слот не хранит данные дольше, чем необходимо. Итерационный счётчик `iteration` позволяет отличать данные текущей итерации от устаревших, если RX по какой-то причине проснулся поздно.
