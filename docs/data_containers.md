# Контейнеры данных

## Обзор

Система контейнеров данных — это **типобезопасная обёртка** над сырым host/device memory с полиморфной семантикой. Она позволяет модулям обмениваться данными независимо от их реального типа (`float`, `cuComplex`, `uint8_t`) и местоположения (CPU/GPU).

Ключевые принципы:
- **Память выделяется только в конструкторе/`reserve()`**
- **`copy()` создаёт глубокую копию** (через `cudaMemcpy` для GPU)
- **Передача между модулями через `std::shared_ptr<IData>`**
- **Инкапсуляция CUDA-памяти** — `cudaMalloc`/`cudaFree` скрыты внутри класса

## Базовый интерфейс IData

`IData` определяет универсальный контракт для всех контейнеров. Это абстрактный базовый класс, который позволяет Conveyor передавать данные между модулями, не зная их конкретного типа.

| Метод | Назначение |
|---|---|
| `size()` | Логический размер данных **в элементах** (не в байтах) |
| `availableSize()` | Зарезервированный/выделенный размер в элементах (ёмкость буфера) |
| `reserve(size)` | **Единственный метод выделения/перевыделения памяти** |
| `isValid()` | Можно ли использовать данные (память выделена, указатель не `nullptr`) |
| `copy()` | Глубокая копия → возвращает `std::shared_ptr<IData>` |
| `getDataName()` | Имя канала логирования/идентификации контейнера |

Ключевое правило: *«Память выделяется только в конструкторе»* (и в `reserve()`). Все контейнеры содержат `boost::log` logger с channel, равным имени контейнера. Это удобно для отладки: логи каждого контейнера можно фильтровать отдельно.

Почему `size()` возвращает количество элементов, а не байтов? Потому что модули работают с семантикой сигнала: «1024 отсчёта float», а не «4096 байт». Это устраняет постоянные деления и умножения на `sizeof(T)` в коде модулей. Если модулю нужен размер в байтах, он может вычислить его через `size() * elementSizeBytes()`.

Почему `copy()` возвращает `shared_ptr<IData>`, а не сырой указатель? Потому что копия должна быть самостоятельным объектом с автоматическим управлением памятью. `shared_ptr` гарантирует, что память освободится, когда ни один модуль больше не ссылается на данные. Это критически важно для GPU-памяти, где утечка стоит дорого.

## Интерфейс GPU-данных IGpuSignalData

`IGpuSignalData` наследует `IData` и добавляет runtime-типизацию для GPU-контейнеров. Это позволяет модулям работать с GPU-данными, не зная конкретного шаблонного типа `T`, до момента запуска CUDA-ядра.

| Метод | Назначение |
|---|---|
| `sampleType()` | Возвращает `GpuSampleType` (`Float32`, `ComplexFloat32`, `UInt8`) |
| `deviceDataRaw()` | `void*` — сырой указатель на device-память (для generic-кода) |
| `elementSizeBytes()` | Размер одного элемента в байтах (`sizeof(T)`) |
| `setLogicalSize(size)` | Устанавливает логический размер **без** перевыделения (если `size ≤ availableSize`) |

Почему `deviceDataRaw()` возвращает `void*`? Потому что generic-код (например, логирование или валидация) не должен зависеть от шаблонного параметра. Конкретный модуль, который знает, что работает с `float`, делает `static_cast<float*>(deviceDataRaw())` перед передачей в CUDA-ядро. `sampleType()` позволяет проверить корректность этого приведения в runtime.

`setLogicalSize(size)` — важная оптимизация. Он позволяет уменьшить видимый размер данных без перевыделения памяти. Например, после декимации количество отсчётов уменьшилось, но буфер на GPU остался прежним. Модуль просто вызывает `setLogicalSize(newSize)`, и следующий модуль увидит корректный размер. Это избегает лишних `cudaFree`/`cudaMalloc`.

## Шаблонный класс GpuSignal<T, Tag>

`GpuSignal` — основной контейнер для GPU-данных. Он шаблонизирован по типу элемента `T` и тегу `Tag` (используется для именования типа в логах и метаданных). CRTP-подобный паттерн с `GpuSampleTypeTraits<T>` маппит C++-тип на `GpuSampleType`.

### Внутреннее устройство

Контейнер хранит четыре поля: логический размер `m_size`, зарезервированный размер `m_resedSize`, флаг валидности `m_isValid` и device-указатель `m_data` (выделен через `cudaMalloc`).

### Конструкторы

- `GpuSignal()` — пустой, невалидный. Удобен как placeholder до первого `reserve()`.
- `GpuSignal(size_t size)` — выделяет через `cudaMalloc(sizeof(T) * size)`. Сразу валиден.
- Move-конструктор/оператор — **перемещает** владение указателем, обнуляет источник. Это предотвращает double-free.
- **Копирование запрещено** (`= delete`). Копирование GPU-памяти — дорогая операция, и запрет на неявное копирование защищает от случайных `cudaMemcpy` в конструкторах.

### Ключевые методы

**`reserve(size)`** — центральный метод управления памятью. Если `size == 0`, он освобождает память. Если `size ≤ m_resedSize`, он **не перевыделяет** (guard от down-malloc), а просто помечает контейнер валидным. Иначе вызывает `cudaFree` + `cudaMalloc` нового блока.

Эта логика крайне эффективна для конвейерной обработки. Представим, что FIR-фильтр обрабатывает блоки по 4096 отсчётов. На первой итерации `reserve(4096)` выделит память. На всех последующих итерациях `reserve(4096)` просто вернёт `true`, не трогая CUDA. Это избавляет от накладных расходов на `cudaMalloc`/`cudaFree` в горячем цикле.

**`setDataFromHost(T* data, size)`** — копирует данные из host-памяти на device через `cudaMemcpyHostToDevice`. Проверяет, что `size ≤ m_resedSize`, чтобы не выйти за границы буфера.

**`setDataFromDevice(T* data, size)`** — копирует данные между device-буферами через `cudaMemcpyDeviceToDevice`. Используется, например, при копировании через `VirtualTransmitter`.

**`setLogicalSize(size)`** — устанавливает `m_size` без перевыделения. `size` не может превышать `m_resedSize`.

**`getDeviceData()`** — возвращает типизированный `T*` (const и non-const). Используется для передачи в CUDA-ядра.

**`copy()`** — создаёт глубокую копию: выделяет новый буфер через `cudaMalloc` и копирует содержимое через `cudaMemcpyDeviceToDevice`. Возвращает `shared_ptr<IData>`.

### Поддерживаемые типы

| Тип C++ | Alias | GpuSampleType |
|---|---|---|
| `float` | `GpuFloatSignal` | `Float32` |
| `cuComplex` | `GpuComplexFloatSignal` | `ComplexFloat32` |
| `uint8_t` | `GpuByteSignal` | `UInt8` |

## CPU-контейнеры

### CpuFloatSignal / CpuComplexSignal

Эти контейнеры наследуют `IData` напрямую (не `IGpuSignalData`), потому что данные хранятся в host-памяти (`new[]`/`delete[]`), а не на GPU.

| Метод | Описание |
|---|---|
| `getData()` | Возвращает `float*` / `cuComplex*` |
| `fromGpu(shared_ptr<IData>)` | **Статическая фабрика**: делает `dynamic_pointer_cast` к соответствующему `Gpu*Signal`, затем `cudaMemcpyDeviceToHost` в `new T[...]` |
| `reserve(size)` | Всегда возвращает `true` (CPU-контейнер не управляет динамическим резервированием; память передаётся извне) |
| `copy()` | Глубокая копия через `std::memcpy` в `new T[m_size]` |

**Важно:** конструкторы принимают **уже выделенный** указатель (`float* data, size_t size`), а контейнер берёт на себя владение (`delete[]` в деструкторе). Это позволяет создавать CPU-контейнер из буфера, выделенного внешней библиотекой (например, из файла через `mmap`), не копируя данные при создании.

`fromGpu()` — мост между мирами GPU и CPU. Модули обычно работают на GPU, но иногда нужно выгрузить результат для анализа, записи в файл или передачи по сети. `fromGpu()` делает это централизованно: проверяет тип через `dynamic_pointer_cast`, выделяет CPU-буфер нужного размера и выполняет `cudaMemcpyDeviceToHost`. Модулю не нужно помнить размеры и типы — всё происходит автоматически.

## EmptyContainer

`EmptyContainer` — заглушка-«нулевой объект». Он реализует `IData`, но все его методы возвращают нейтральные значения: `size()` — 0, `availableSize()` — 0, `reserve(...)` — `false`, `isValid()` — `false`, `copy()` — новый `EmptyContainer`.

Это удобно, потому что избавляет от проверок на `nullptr` в Conveyor. Вместо того чтобы спрашивать «а есть ли данные?», Conveyor просто вызывает `size()` и сравнивает с нулём. Если модуль вернул `EmptyContainer`, Conveyor корректно завершает конвейер. `VirtualTX` отправляет `EmptyContainer` при уничтожении как сигнал завершения — получатели узнают, что новых данных больше не будет.

## Зачем обёртки, а не сырые указатели

### 1. Полиморфная передача через `std::shared_ptr<IData>`

Модули в конвейере общаются через единый интерфейс:

```cpp
void setData(std::shared_ptr<IData> data);
std::shared_ptr<IData> getData();
```

Модуль может принимать `float`, `complex` или `uint8_t` — не зная заранее, что ему придёт. Он работает с `IGpuSignalData` и `GpuSampleType` для runtime-типизации. Это удобно, потому что позволяет строить универсальные модули. Например, `FileWriter` может записывать любой тип данных: он просто спрашивает `sampleType()` и `size()`, а затем вызывает `cudaMemcpyDeviceToHost` с правильным размером элемента.

### 2. Глубокое копирование (deep copy) для VirtualTransmitter

`VirtualTransmitter` рассылает данные нескольким RX-получателям. Каждый получатель должен иметь **независимую** копию. Метод `copy()` создаёт глубокую копию:
- Для GPU: `cudaMalloc` + `cudaMemcpyDeviceToDevice`
- Для CPU: `new[]` + `std::memcpy`

Тесты (`data_container_copy_tests.cpp`) явно проверяют: *«Изменение копии не влияет на оригинал»*.

Без обёртки модуль должен был бы сам вызывать `cudaMalloc` и `cudaMemcpy`, зная размер и тип данных. С обёрткой он просто вызывает `data->copy()`, и всё происходит автоматически. Это устраняет дублирование кода и снижает вероятность ошибок с размерами.

### 3. Типобезопасность + runtime-типизация

`GpuSampleType` + `GpuSampleTypeTraits<T>` позволяют generic-коду понимать, с чем он работает, без шаблонов. `validateGpuInput()` в `GpuSignalUtils.hpp` централизованно проверяет: не `nullptr`, валиден, допустимый тип, достаточный размер. `dynamic_pointer_cast` используется для строгой типизации при CPU↔GPU конвертации.

Это эффективно, потому что валидация происходит один раз — при получении данных модулем. Если данные невалидны, модуль сразу возвращает `false`, и Conveyor останавливает конвейер. Без обёртки каждый модуль проверял бы указатели самостоятельно, что привело бы к дублированию guard-кода.

### 4. Инкапсуляция управления CUDA-памятью

`GpuSignal` полностью скрывает `cudaMalloc` / `cudaFree`:
- Деструктор вызывает `freeData()` → `cudaFree`
- Move-семантика предотвращает double-free
- `reserve()` реализует оптимизацию «не перевыделять, если влезает» (buffer reuse)

Это критически важно для надёжности. CUDA-память — ограниченный ресурс, и утечка в одном модуле может привести к `cudaErrorMemoryAllocation` во всех остальных. Инкапсуляция гарантирует, что память освободится ровно один раз — при уничтожении последнего `shared_ptr`. Модуль не может случайно вызвать `cudaFree` дважды или забыть освободить буфер при исключении.

### 5. Единое правило жизненного цикла

`IData` гарантирует, что память выделяется только в конструкторе/`reserve()`. Это упрощает отладку утечек и делает поведение предсказуемым. Если контейнер валиден, у него есть память. Если невалиден — памяти нет. Нет промежуточных состояний вида «указатель есть, но размер неинициализирован».

## Утилиты: GpuSignalUtils

`GpuSignalUtils` (`DataContainers/include/GpuSignalUtils.hpp`) содержит вспомогательные inline-функции для модулей. Они не являются методами класса, потому что работают с интерфейсом `IData`, а не с конкретным `GpuSignal<T>`.

| Функция | Назначение |
|---|---|
| `asGpuSignal(data)` | Безопасный `dynamic_pointer_cast<IGpuSignalData>` |
| `validateGpuInput(data, allowedTypes, minSize, context)` | Цепочка проверок: `nullptr` → `isValid()` → тип → размер → указатель |
| `createLike(prototype, size)` | Фабрика: создаёт GPU-контейнер того же `GpuSampleType`, что и `prototype` |
| `ensureHistoryLike(input, historySize, history, nextHistory)` | Управление буферами истории для скользящей обработки (например, FIR): создаёт/переиспользует буферы, обнуляет `cudaMemset` |

`createLike()` особенно полезен. Представим, что FIR-фильтр получает входной сигнал `GpuFloatSignal` и должен создать выходной буфер того же типа. Вместо ручного выбора между `GpuFloatSignal`, `GpuComplexFloatSignal` и `GpuByteSignal`, модуль просто вызывает `createLike(input, outputSize)`. Утилита смотрит `sampleType()` прототипа и создаёт контейнер нужного типа. Это устраняет switch-case по типам в каждом модуле.

`ensureHistoryLike()` решает проблему истории для FIR-фильтров. FIR требует N-1 предыдущих отсчётов для вычисления текущего. Модуль передаёт текущий входной буфер и буфер истории, а утилита либо создаёт новый буфер истории нужного размера, либо переиспользует существующий. После обработки указатели `history` и `nextHistory` меняются местами — классический double-buffering. Обнуление через `cudaMemset` гарантирует, что на первой итерации история заполнена нулями.

## Передача данных между модулями

Передача данных внутри конвейера происходит в пять шагов, и ни один из них не требует копирования памяти.

Сначала модуль `A` создаёт или обрабатывает данные, упаковывая результат в `std::shared_ptr<IData>` (например, `GpuFloatSignal`). Затем `Conveyor` вызывает `A->getData()` и получает `shared_ptr<IData>`. Conveyor проверяет валидность и размер, после чего передаёт тот же `shared_ptr` в модуль `B` через `B->setData(data)`. Модуль `B` приводит тип через `asGpuSignal(data)` → `validateGpuInput(...)` → `deviceDataRaw()` / `getDeviceData()`. После обработки `B` возвращает новые данные через свой `getData()`, и цикл повторяется.

Для межконвейерной передачи (`VirtualTX` → `VirtualRX`) `VirtualTransmitter` делает `copy()`, чтобы каждый RX получил независимый экземпляр. Это единственное место, где происходит копирование — и оно оптимизировано (последний получатель забирает оригинал через `move`).

Такая архитектура эффективна, потому что внутри конвейера данные передаются по ссылке (через `shared_ptr`), а копирование происходит только при реальной необходимости — межконвейерной рассылке. GPU-память остаётся на GPU на всём протяжении обработки, не поднимаясь на CPU.
