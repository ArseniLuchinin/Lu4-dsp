### Здесь описывается архитектура приложения

## Компоненты
- **Core** — управление конвейерами обработки и создание модулей.
- **DataContainers** — контейнеры для хранения данных на CPU и GPU, которыми манипулируют модули.
- **Module** — базовые интерфейсы и инструменты для создания модулей.
- **Modules** — набор готовых модулей обработки.
- **Utils** — вспомогательные скрипты для генерации сигналов и шаблоны.

## Конфигурация
Выполнение расчёта задаётся двумя файлами. Это нужно, чтобы можно было менять переменные без изменения конфига, например для разных сигналов.
- **pipeline.json** — главный конфиг: определяет конвейеры (наборы модулей), их порядок и параметры модулей. Поддерживает подстановку переменных через $VAR_NAME.
- **variables.toml** — глобальные переменные используемые в pipeline.json.

## Описание архитектуры выполнения расчёта

1. В модули передаются параметры (`setParam`)
2. Конвейер инициализирует модули (`init`). Если все модули инициализированы, начинается расчёт.
   - При инициализации модуль проверяет корректность параметров.
3. Первым делом конвейер всегда запускает первый модуль (`run`), так как в него не передаются другие данные.
4. Проверяется, отработал ли модуль без ошибок.
5. Проверяется валидность данных на выходе (`isValid`). Если данные не валидны, работа прекращается.
6. Данные передаются в следующий модуль (`setData`). Если модуль совместим с этими данными, конвейер продолжает работу.
   - При получении данных модуль проверяет совместимость типа входных данных и обычно проверяет `isValid`.
   - **Важно:** модуль может как создавать новый контейнер для данных, так и переписать старый — это никак не учитывается и решается на усмотрение разработчика модуля.
7. Начинается обработка данных (`run`). Если модуль отработал корректно, конвейер продолжает работу.
    - При расчёте модуль может делать guard-проверки (например, на `nullptr` и обязательные зависимости), даже если базовая валидация уже сделана в `init` и `setData`.
8. Повторяются пункты 4-7.

## Описание интерфейсов модуля

### IModule

Каждый модуль реализует интерфейс `IModule`, который определяет следующие методы:

- `init()` — инициализация модуля. Вызывается один раз перед началом расчёта. Здесь проверяются параметры и подготавливаются внутренние структуры данных.
- `run()` — запуск обработки данных. Основная логика модуля выполняется здесь.
- `setParam(paramName, value)` — установка параметров модуля. Параметры передаются из `pipeline.json` через этот метод.
- `setData(data)` — получение и проверка входных данных от предыдущего модуля. Тип данных должен быть совместим с модулем.
- `getData()` — возврат обработанных данных для передачи следующему модулю.

### VirtualTransmitter

Механизм передачи данных между модулями, работающими в разных конвейерах или потоках. Использует именованные теги для идентификации данных.

- `txData(data, tag)` — отправка данных с указанным тегом.
- `rxData(tag)` — получение данных по тегу (без ожидания).
- `waitRxData(tag)` — получение данных с ожиданием (блокирует поток, пока данные не появятся).
- `checkData(tag)` — проверка наличия данных с указанным тегом.

Модули `VirtualTX` и `VirtualRX` используют этот механизм для связи между конвейерами.

### IVirtualRX

Интерфейс для модулей, принимающих данные через `VirtualTransmitter`. Позволяет указать тег, по которому модуль будет получать данные.
